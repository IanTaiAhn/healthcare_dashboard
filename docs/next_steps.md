1. Get a FRED API key. Go to https://fred.stlouisfed.org/docs/api/api_key.html, create an account (or sign in), and request a key. It's free and instant — you get a 32-character hex string.
2. Add it to your .env file. Open backend/.env and replace your_fred_api_key_here with the actual key.
3. Uncomment two lines in backend/data/ingestion.py. Right now the fredapi import at the top is commented out, and get_fred_client() raises NotImplementedError instead of returning a client. You need to uncomment from fredapi import Fred at the top and the return Fred(api_key=api_key) line inside get_fred_client(). Then do the same for fetch_series() — uncomment the two lines that call fred.get_series() and remove the raise NotImplementedError.
4. Test it by running notebook 01. Open notebooks/01_data_exploration.ipynb, switch from generate_dummy_series() to fetch_all_series(), and run it. This will attempt to pull every series in preprocessing_config.yaml from FRED. Some might fail — a series ID could be wrong, or the API might rate-limit you if you hit it too fast. The notebook will print which ones succeeded and which failed, so you can debug individual series.
5. Work through notebooks 02–04 in order. Notebook 02 runs the ADF stationarity tests and gives you the adf_pvalue values to write back into preprocessing_config.yaml. Notebook 03 tells you whether the features actually predict the target. Notebook 04 fits the models and picks the winner.
6. Wire the API routes to real data. Once you have a trained model saved as sarimax_current.joblib, you replace the DUMMY_EMPLOYMENT_FORECAST dict in backend/api/routes/forecast.py with actual model inference — load the model from app.state.model and call the forecast function. Same pattern for the signal and indicator routes: replace the dummy dicts with calls to the real preprocessing and engineering functions.
The key thing is that steps 1–5 happen entirely in notebooks before you touch any of the production API code. The notebooks are your sandbox for validating that the data is clean and the model actually works. You only wire the API routes to real data after you're confident in what the pipeline produces.
